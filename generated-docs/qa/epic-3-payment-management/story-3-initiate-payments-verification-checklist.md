# Manual Verification Checklist — Epic 3, Story 3: Initiate Payments Button and Batch Creation

> Generated by WRITE-TESTS agent | 2026-05-14
> Reviewed during QA phase before sign-off.
> Automated Playwright spec (`web/e2e/epic-3-story-3-initiate-payments.spec.ts`) must pass first.

**Route:** `/payment-management`
**Prerequisite:** Sign in as admin (`admin@example.com` / `Admin123!`).

---

## Button Enabled/Disabled State

- [ ] **AC-1** — Open the Payment Management page. The AgencyName filter shows "All Agencies" by default. The "Initiate Payments" button is visibly disabled (greyed out, not clickable).
- [ ] **AC-2** — Select an agency from the dropdown that has **zero** REG-status payments (all MAN-PAY or PARKED). Verify the "Initiate Payments" button remains disabled.
- [ ] **AC-3** — Select an agency that has at least one REG-status payment. Verify the "Initiate Payments" button becomes enabled (clickable, not greyed out).
- [ ] **AC-18** — While the button is disabled, inspect it in DevTools. Verify it has either a native `disabled` attribute or `aria-disabled="true"` so screen readers announce the disabled state.

---

## Payload and Header (Network Tab Verification)

- [ ] **AC-4, AC-5** — With an agency selected that has REG payments, open the browser Network tab. Click "Initiate Payments". Verify a `POST /v1/payment-batches` request is sent with body `{ "PaymentIds": [<REG IDs only>] }`.
- [ ] **AC-6** — In the same Network request, confirm that no MAN-PAY payment IDs are included in `PaymentIds`.
- [ ] **AC-7** — Confirm that no payment IDs from other agencies are included in `PaymentIds` — only IDs belonging to the currently selected agency with REG status.
- [ ] **AC-8** — In the request headers, verify `LastChangedUser: <your session username>` is present (e.g. `LastChangedUser: Admin User`).

---

## In-Flight Loading Guard (BA-2)

- [ ] **AC-21** — Click "Initiate Payments" and immediately observe the button. While the `POST /v1/payment-batches` call is in flight, the button must be visibly disabled and show a loading indicator (spinner or similar). A second click during this period must not submit a second request.
- [ ] **AC-22 (success)** — After the batch creation call completes successfully, the button returns to its enabled or disabled state based on the refreshed payment data (if no REG payments remain, it goes disabled).
- [ ] **AC-22 (failure)** — After a failed call, the button returns to enabled (if the agency still has REG payments) so the user can retry.

---

## Success Flow

- [ ] **AC-10** — After a successful `POST /v1/payment-batches`, a toast notification appears with the exact text: **"Payment batch created successfully."**
- [ ] **AC-11** — The payment grid automatically refreshes (`GET /v1/payments` is called again). Previously REG payments for the selected agency should no longer show REG status (or may disappear from the filtered view). All row checkboxes are cleared after the refresh.
- [ ] **AC-12** — The page URL remains `/payment-management` after the success. The user is NOT automatically navigated to another page.
- [ ] **AC-15** — The success toast auto-dismisses after approximately 5 seconds without user interaction.
- [ ] **AC-16** — The success toast can be dismissed immediately by clicking its close/dismiss control.

---

## Failure Flow

- [ ] **AC-13** — Force a failure (e.g., stop the backend, or use browser DevTools to block `POST /v1/payment-batches`). Click "Initiate Payments". Verify a toast appears with the exact text: **"Failed to initiate payment batch. Please try again."**
- [ ] **AC-14** — After the failure, the "Initiate Payments" button is enabled again (assuming the selected agency still has REG payments). The user can retry.
- [ ] **AC-15** — The failure toast auto-dismisses after approximately 5 seconds.
- [ ] **AC-16** — The failure toast can be dismissed immediately by clicking its close/dismiss control.

---

## Session Guard (AC-9)

- [ ] **AC-9** — Sign out (or expire the session by clearing cookies). Navigate back to `/payment-management`. If the page is visible and the session username is unavailable, clicking "Initiate Payments" must NOT call the API. Instead, the user should be redirected to `/auth/signin`.

---

## Keyboard Accessibility

- [ ] **AC-19** — Using only the keyboard (Tab key), navigate to the "Initiate Payments" button. Verify the button receives a visible focus indicator (ring or outline) that is clearly distinguishable from the non-focused state.
- [ ] **AC-20 (Enter)** — With the button focused and enabled, press **Enter**. Verify it triggers the same action as a mouse click (batch creation call is made, toast appears).
- [ ] **AC-20 (Space)** — With the button focused and enabled, press **Space**. Verify it triggers the same action.

## Data-Contract Checks (Runtime Verification — from Handoff)
