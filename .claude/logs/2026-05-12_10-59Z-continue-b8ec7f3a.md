I read through the workflow scripts and agent specs, and I want to share some honest concerns before designing this change.

## What I found

The current workflow is **architecturally enforced** as per-story-sequential, not just by convention:

1. **`transition-phase.js:192-199`** explicitly blocks `Story N` from entering REALIGN/WRITE-TESTS until `Story N-1` is `COMPLETE`. The state machine is the gatekeeper.
2. **`developer.md:126-145`** â€” the IMPLEMENT agent scans `web/src/` to see what prior stories already built. If Story 1.1's code doesn't exist yet, Story 1.2's IMPLEMENT can't tell what foundation it should reuse vs. build â€” it'll either duplicate or fail.
3. **`test-generator.md:581-588`** â€” Story 1 renders full pages; Story 2+ renders components in isolation **because** Story 1 already wired the page. Writing Story 1.2's tests before 1.1 is implemented loses that scoping signal.
4. **Dashboard state** tracks `currentStory` as a single scalar, not an array of in-flight stories.
5. **QA manual verification is inherently per-story** â€” you have to actually click around in the browser and check each story's ACs, which requires the implementation to exist.

## My honest take

I don't think batching the **whole** pipeline by phase is a good trade. You'd save coordination time but lose the TDD feedback loop â€” the thing that catches "Story 1.2's tests assumed Story 1.1 would do X, but the implementation actually did Y" gets discovered 5 stories later instead of immediately. The QA fix cycle, spec-drift detection, and per-story commit traceability all assume the sequential model.

**However**, there's a real, narrower optimization that gets most of the benefit:

> **Batch only the *human-gating* phases (REALIGN approvals + TEST-DESIGN BA decisions) for all stories in an epic upfront. Then run WRITE-TESTS â†’ IMPLEMENT â†’ QA per story sequentially.**

REALIGN and TEST-DESIGN don't touch code â€” they just refine acceptance criteria and surface BA decisions. Batching the AskUserQuestion prompts across all 3 stories of an epic into a single review session would:
- Eliminate 2 of the 3 user-interrupt points per story (you only get interrupted for QA verification, which **has** to be per-story because the code has to exist)
- Reduce orchestrator round-trips
- Keep the safety properties (implementation feedback, per-story commits, spec drift detection) intact

Other lower-risk options worth considering:
- **Skip TEST-DESIGN entirely for stories with zero BA decisions** (e.g., Story 1.3 â€” pure formatting utilities have no ambiguous business logic)
- **Combine trivial stories at SCOPE time** instead of running 3 stories where 2 would do
- **Auto-approve REALIGN when the agent reports "no impacts"** (already partially handled per `orchestrator-rules.md`)
